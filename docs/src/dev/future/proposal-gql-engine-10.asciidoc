*Proposal 10*

== Generalized GQL Execution Engine

=== Introduction

The `match(String)` step introduced in TinkerPop 4.0 allows declarative GQL pattern matching to
be embedded inside a Gremlin traversal. The current implementation of this engine lives entirely
inside `tinkergraph-gremlin` and is coupled to TinkerGraph's internal data structures. This means
that every graph database that wants to offer a working `match()` step must implement the full GQL
stack from scratch: grammar, parser, query planner, and DFS executor.

This proposal describes extracting that engine into a new, independently usable Maven module —
`gql-gremlin` — so that any TinkerPop-compliant graph database can adopt it as a dependency and
get a working `match()` implementation with minimal effort. Providers that prefer to supply their
own native implementation are unaffected; the module is strictly opt-in.

=== Motivation

At the outset of building `match()`, it was not clear what form the GQL execution engine would take.
As it developed in TinkerGraph, however, the engine started to form with relatively clear lines for
where the planner and executor could consult the graph for information to make better decisions. Any
`Graph` implementation can already offer GQL using only the standard `Graph`, `Vertex`, and `Edge`
APIs, but without cardinality hints and index access the engine falls back to full vertex scans and
arbitrary seed selection — potentially painful on large graphs.

Offering this engine as a default reference implementation lowers the bar for providers to offer
basic GQL support. It spares providers the need to develop and maintain their own grammar and parser,
come up with their own query planner and execution engine, or handle other related details. As an
added benefit for users, the TinkerPop form of GQL offers some level of standardization about what
TinkerPop supports with `match` so that the feature has some potential to carry from one graph to
another.

==== Why a Separate Module, Not `gremlin-core`

The step _contract_ (return type, path label recording, `TraverserRequirement.LABELED_PATH`) belongs
in `gremlin-core` and is already there via `DeclarativeMatchStep`. The step _implementation_
is a different concern.

Placing the execution engine in `gremlin-core` would impose the ANTLR4 GQL grammar, the query
model classes, and the planner/executor on every TinkerPop consumer regardless of whether they
use `match()`. While `gremlin-core` already depends on ANTLR4 for the Gremlin grammar, that grammar
is fundamental to the traversal engine itself. The GQL grammar is optional infrastructure for one
step. Keeping them separate maintains a clean distinction.

A dedicated `gql-gremlin` module also gives the GQL grammar and executor room to evolve at their
own pace — adding new syntax, improving the planner, introducing property filter pushdown — without
touching `gremlin-core`'s compatibility guarantees.

The `gremlin-test` and `gremlin-tools` modules set the precedent: optional, composable pieces live
in their own modules. `gql-gremlin` follows that pattern.

=== Architecture

The following diagram shows the intended module responsibility split.

----
gremlin-core
  └── DeclarativeMatchStep              (placeholder; step contract only)
  └── GraphTraversalSource.match()      (API; unchanged)
  └── GraphTraversal.match()            (API; unchanged)
  └── Graph.countVerticesByLabel()      (new default method; returns Long.MAX_VALUE)
  └── Graph.countEdgesByLabel()         (new default method; returns Long.MAX_VALUE)
  └── Graph.index()                     (new default method; returns Graph.Index.EMPTY)
  └── Graph.Index                       (new nested interface; queryVertexIndex, countVertexIndex)

gql-gremlin  (new module, depends on gremlin-core)
  ├── GQL.g4                            (ANTLR4 grammar, currently in tinkergraph-gremlin)
  ├── QueryGraph / QueryVertex / QueryEdge  (parsed query model)
  ├── PropertyPredicate / ExtensionStep     (plan model)
  ├── GqlMatchPlan                      (compiled plan)
  ├── GqlPlanner  (interface)           (compiles a query string into a GqlMatchPlan)
  ├── DefaultGqlPlanner                 (calls graph.countVerticesByLabel / countEdgesByLabel)
  ├── GqlExecutor (interface)           (executes a GqlMatchPlan against a Graph)
  ├── DefaultGqlExecutor                (calls graph.index().queryVertexIndex / countVertexIndex)
  ├── GqlMatchStep                      (concrete AbstractStep; wires plan + executor into traversal)
  └── GqlDeclarativeMatchStrategy       (ProviderOptimizationStrategy; registers the above)

tinkergraph-gremlin  (depends on gql-gremlin)
  ├── AbstractTinkerGraph overrides countVerticesByLabel()  (O(1) via label count map)
  ├── AbstractTinkerGraph overrides countEdgesByLabel()     (O(1) via label count map)
  └── AbstractTinkerGraph overrides index()                 (returns TinkerIndex backed by TinkerIndexHelper)
----

A new provider wishing to support `match()` needs to:

. Add `gql-gremlin` as a Maven dependency.
. Register `GqlDeclarativeMatchStrategy` with their `GraphTraversalSource`.
. Optionally override `countVerticesByLabel()`, `countEdgesByLabel()`, and `index()` on their
  `Graph` to unlock performance optimizations.

That is the entire adoption path for a correct, functional implementation.

=== Graph Interface Extensions

Two default methods and one nested interface are added to `Graph` in `gremlin-core`. These follow
the existing pattern of `Graph.Variables` (a nested interface accessed via a default-backed accessor
on `Graph`) and `getServiceRegistry()` (a default method returning a no-op singleton when not
overridden).

==== Cardinality Methods

[source,java]
----
/**
 * Returns the number of vertices with the given label, or Long.MAX_VALUE if unknown.
 * A null label means all vertices regardless of label.
 *
 * The default implementation returns Long.MAX_VALUE. Graph implementations that
 * maintain label cardinality counts should override this method to enable the GQL
 * planner to select an optimal seed variable and order extension steps by label density.
 * Approximate values are sufficient; exact counts are not required.
 */
default long countVerticesByLabel(final String label) {
    return Long.MAX_VALUE;
}

/**
 * Returns the number of edges with the given label, or Long.MAX_VALUE if unknown.
 * A null label means all edges regardless of label.
 *
 * The default implementation returns Long.MAX_VALUE. Graph implementations that
 * maintain edge label cardinality counts should override this method to enable the
 * GQL planner to order extension steps by edge-label density so that rarer labels
 * prune the DFS search space sooner.
 */
default long countEdgesByLabel(final String label) {
    return Long.MAX_VALUE;
}
----

==== Graph.Index

`Graph.Index` is a nested interface modelled after `Graph.Variables`. The accessor `index()` is a
default method returning `Graph.Index.EMPTY`, a no-op singleton whose methods return conservative
values. Graphs that maintain property indexes override `index()` to return a live implementation.

[source,java]
----
/**
 * Returns the Index accessor for this graph. The default implementation returns
 * Graph.Index.EMPTY, whose methods return conservative values (empty iterator /
 * Long.MAX_VALUE) indicating that no property index is available.
 *
 * Graph implementations that maintain property indexes should override this method
 * and return an Index backed by their index structures. The GQL executor uses the
 * Index to replace full vertex scans with targeted lookups for selective predicates.
 */
default Index index() {
    return Index.EMPTY;
}

/**
 * Provides access to property index lookups for the graph. Analogous to Graph.Variables
 * for graph-level metadata, Graph.Index exposes the read side of a graph's property
 * index structures.
 *
 * Implementations whose graphs do not support property indexes need not implement this
 * interface; the default Graph.index() accessor returns Graph.Index.EMPTY.
 */
interface Index {

    /**
     * Returns an iterator over all vertices where the given property key equals the
     * given value. Must return an empty iterator (not null) when no match exists or
     * when the key is not indexed.
     */
    default Iterator<Vertex> queryVertexIndex(final String key, final Object value) {
        return Collections.emptyIterator();
    }

    /**
     * Returns the number of vertices where the given property key equals the given
     * value. Returns Long.MAX_VALUE when the key is not indexed or the count is
     * unknown. Used by the GQL planner to select the most selective index predicate
     * when multiple indexed keys appear in the same node filter.
     */
    default long countVertexIndex(final String key, final Object value) {
        return Long.MAX_VALUE;
    }

    /** No-op singleton returned by the default Graph.index() implementation. */
    Index EMPTY = new Index() {};
}
----

Note that `Graph.Index` is intentionally vertex-only. Edge index access may be added in a future
revision as the engine gains edge-first seed selection support.

=== GqlPlanner and GqlExecutor

These interfaces allow providers to substitute their own implementations without replacing the full
strategy, while `DefaultGqlPlanner` and `DefaultGqlExecutor` call the `Graph` methods unconditionally
— no `instanceof` checks or adapters are needed.

[source,java]
----
public interface GqlPlanner {
    /**
     * Compiles a GQL MATCH string into an executable plan. Implementations are
     * encouraged to cache the parsed QueryGraph keyed on the query string; seed
     * selection should be recomputed on each call so that graph mutations are
     * reflected in subsequent executions.
     */
    GqlMatchPlan plan(String gqlMatchString);
}

public interface GqlExecutor {
    /** Executes the plan with no parameter bindings. */
    Iterator<Element[]> execute(GqlMatchPlan plan);

    /** Executes the plan with the given parameter bindings. */
    Iterator<Element[]> execute(GqlMatchPlan plan, Map<String, Object> params);
}
----

`DefaultGqlPlanner` takes a `Graph` and calls `graph.countVerticesByLabel()` and
`graph.countEdgesByLabel()` directly for seed selection and step ordering. `DefaultGqlExecutor` takes
a `Graph` and calls `graph.index().queryVertexIndex()` and `graph.index().countVertexIndex()` for
seed iteration, falling back to a full scan when `graph.index()` returns `Graph.Index.EMPTY`.

=== GqlDeclarativeMatchStrategy

[source,java]
----
/**
 * ProviderOptimizationStrategy that replaces DeclarativeMatchStep with a concrete
 * GqlMatchStep backed by the supplied planner and executor.
 *
 * Providers that want the reference engine register this strategy directly:
 *
 *   graph.traversal()
 *        .withStrategies(GqlDeclarativeMatchStrategy.create(graph))
 *
 * The static factory constructs DefaultGqlPlanner and DefaultGqlExecutor over the
 * supplied graph. Both delegate to the graph's countVerticesByLabel(), countEdgesByLabel(),
 * and index() methods, which return conservative defaults unless the graph overrides them.
 */
public class GqlDeclarativeMatchStrategy extends AbstractTraversalStrategy<...>
        implements ProviderOptimizationStrategy {

    public static GqlDeclarativeMatchStrategy create(Graph graph) { ... }

    public static GqlDeclarativeMatchStrategy create(GqlPlanner planner,
                                                     GqlExecutor executor) { ... }
}
----

=== Graceful Degradation

The reference engine degrades gracefully based on which `Graph` methods are overridden:

[width="100%",cols="3,3,4",options="header"]
|=========================================================
|`countVerticesByLabel` / `countEdgesByLabel` |`index()` |Behavior
|Default (`Long.MAX_VALUE`) |Default (`EMPTY`) |Full vertex scan; seed chosen arbitrarily; step order is parse order
|Overridden |Default (`EMPTY`) |Full vertex scan; seed is lowest-cardinality node; steps ordered by edge density
|Default (`Long.MAX_VALUE`) |Overridden |Index lookup for seed; step order is parse order
|Overridden |Overridden |Index lookup for seed; lowest-cardinality index predicate chosen; steps ordered by edge density
|=========================================================

In all cases the results are correct. The degraded modes are simply slower on large graphs.

=== TinkerGraph Migration

The migration is largely mechanical:

. Add `countVerticesByLabel()`, `countEdgesByLabel()`, and `index()` as default methods to `Graph`
  in `gremlin-core`. Add the `Graph.Index` nested interface.
. Move `GQL.g4`, `QueryGraph`, `QueryVertex`, `QueryEdge`, `PropertyPredicate`,
  `ExtensionStep`, and `GqlMatchPlan` from `tinkergraph-gremlin` to `gql-gremlin` unchanged.
. Extract `GqlPlanner` and `GqlExecutor` interfaces and move the `Default*` implementations
  into `gql-gremlin`.
. Move `GqlMatchStep` and `GqlDeclarativeMatchStrategy` into `gql-gremlin`.
. In `tinkergraph-gremlin`:
** Override `countVerticesByLabel()` and `countEdgesByLabel()` on `AbstractTinkerGraph` using the
   existing O(1) label count maps.
** Override `index()` on `AbstractTinkerGraph` to return a `TinkerIndex` implementation backed by
   `TinkerIndexHelper`.
** Remove `TinkerGraphGqlPlanner` and `TinkerGraphGqlExecutor` — they are no longer needed since
   `DefaultGqlPlanner` and `DefaultGqlExecutor` call the graph methods directly.
** Update `TinkerGraphDeclarativeMatchStrategy` to simply delegate to
   `GqlDeclarativeMatchStrategy.create(graph)`, or remove it entirely if TinkerGraph
   auto-registers the strategy at traversal source construction time.
. Add `gql-gremlin` as a `compile` dependency of `tinkergraph-gremlin`.
. Convert TinkerGraph `match()` tests to Gherkin feature tests in `gremlin-test`.
